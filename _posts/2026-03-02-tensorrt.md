This project built a TensorRT-based inference pipeline targeting NVIDIA T4 and edge deployment. The goal was to maximize throughput for LLM and CNN inference under production latency constraints. The final pipeline achieved 3× throughput over the FP32 baseline and an additional 15% end-to-end latency reduction through custom C++/CUDA plugins for activation functions not natively supported by TensorRT.

## Background: Quantization, Layer Fusion, and TensorRT

**INT8 quantization** reduces model weights and activations from 32-bit or 16-bit floating point to 8-bit integers. NVIDIA GPUs have dedicated INT8 Tensor Cores that deliver significantly higher throughput than FP32 units — on the T4, INT8 delivers 65 TOPS vs. 8.1 TFLOPS for FP32 [1]. The practical inference speedup is typically 2–4× after accounting for memory bandwidth and overhead.

The challenge is precision. INT8 has only 256 representable values, versus the continuous range of FP32. Naively rounding all weights and activations to INT8 typically degrades model accuracy. Post-Training Quantization (PTQ) calibrates INT8 scales by running a representative dataset through the FP32 model and observing the activation distributions, then selecting INT8 scale factors that minimize quantization error [2]. TensorRT automates PTQ but exposes a calibration interface for custom strategies.

**Layer fusion** is a separate optimization: TensorRT's graph optimizer identifies patterns of adjacent operations (e.g., Convolution → BatchNorm → ReLU) that can be executed as a single fused kernel rather than three separate ones [3]. Fusion eliminates intermediate tensor writes to HBM between operations, reducing memory bandwidth usage and kernel launch overhead. On bandwidth-constrained devices like the T4, this matters substantially.

TensorRT's `IPluginV2` interface allows custom CUDA kernels to be integrated directly into a TensorRT execution plan [3]. Plugins define custom operations — implementing `enqueue()` for the CUDA kernel, `getSerializationSize()` and `serialize()` for plan caching, and `getWorkspaceSize()` for temporary memory allocation. This is necessary when a model uses activation functions or custom layers that TensorRT's built-in op registry does not support.

## What We Did

**INT8 PTQ calibration.** We applied TensorRT's INT8 Post-Training Quantization with a representative calibration dataset. The default entropy-based calibrator works well when the calibration distribution matches production inputs. We observed accuracy degradation on long-tail inputs — cases where activation distributions in the calibration set did not cover the tails of the production distribution. We implemented a custom calibrator that sampled from the actual production distribution, weighted to oversample rare but important input types. This recovered accuracy on the edge cases that mattered.

We benchmarked precision vs. accuracy trade-offs systematically across FP32, FP16, and INT8 on both LLM and CNN model families, measuring throughput (requests/second) and accuracy (task-specific metric) at each precision level on the T4.

**Layer fusion.** TensorRT's automatic graph optimizer collapsed Conv-BN-ReLU sequences and other fuse-able patterns during engine build. We measured the throughput contribution of fusion by comparing the full optimized engine against manually disabled fusion on selected layers. Fusion contributed a consistent 25–30% throughput improvement on the T4, consistent with the expected benefit of eliminating intermediate HBM writes [3].

**Custom TensorRT plugins.** The LLM architectures targeted for edge deployment used activation functions not in TensorRT's op registry — specifically modified attention variants with custom gating nonlinearities. We implemented these as `IPluginV2` plugins in C++ and CUDA, integrating them into the TensorRT graph so that the entire model could execute within the TensorRT runtime without falling back to PyTorch for unsupported operations.

A non-trivial challenge was workspace allocation: `IPluginV2` requires declaring the maximum temporary memory needed at plan-build time (a static size), but the activation functions had memory requirements that varied with input sequence length. We resolved this by overallocating based on the maximum supported sequence length and restructuring the kernel to reuse workspace regions sequentially rather than holding them simultaneously. The plugin memory optimization contributed the 15% additional end-to-end latency reduction.

We profiled the full pipeline using Nsight Systems to identify scheduling bottlenecks and CUDA stream underutilization, and used the profile data to configure CUDA stream priorities and kernel overlap for the multi-stage inference pipeline.
