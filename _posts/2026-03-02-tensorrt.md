This project built a TensorRT-based inference pipeline for BERT-base on NVIDIA T4, targeting production latency constraints. The final pipeline achieved **5× throughput over the FP32 baseline** at batch=8, with additional **27% latency reduction at batch=1** through multiple TensorRT optimization profiles, and custom C++/CUDA IPluginV3 plugins for GELU activation.

## Background: Quantization, Layer Fusion, and TensorRT

**INT8 quantization** reduces model weights and activations from 32-bit or 16-bit floating point to 8-bit integers. NVIDIA GPUs have dedicated INT8 Tensor Cores that deliver significantly higher throughput than FP32 units — on the T4, INT8 delivers 65 TOPS vs. 8.1 TFLOPS for FP32 [1]. The practical inference speedup is typically 2–4× after accounting for memory bandwidth and overhead.

The challenge is precision. INT8 has only 256 representable values, versus the continuous range of FP32. Post-Training Quantization (PTQ) calibrates INT8 scales by running a representative dataset through the FP32 model and observing the activation distributions, then selecting INT8 scale factors that minimize quantization error [2]. TensorRT automates PTQ but exposes a calibration interface for custom strategies.

**Layer fusion** is a separate optimization: TensorRT's graph optimizer identifies patterns of adjacent operations that can be executed as a single fused kernel rather than multiple separate ones [3]. For BERT-base, TensorRT reduced ~870 ONNX ops to 201 TRT layers (~4× reduction). Key fusions include QKV MatMul (3→1 per layer across 12 heads), LayerNorm, GELU, Softmax, and scaled dot-product attention.

TensorRT's **IPluginV3** interface (the replacement for the deprecated IPluginV2 in TensorRT 10.x) allows custom CUDA kernels to be integrated directly into a TensorRT execution plan [3]. Plugins implement the IPluginV3OneCore, IPluginV3OneBuild, and IPluginV3OneRuntime capability interfaces, separating build-time and runtime behavior.

## What We Did

### FP16 and INT8 PTQ Calibration

We exported BERT-base from HuggingFace to ONNX (opset 17) and built separate TensorRT engines for FP32, FP16, and INT8 precision tiers. INT8 used TensorRT's IInt8EntropyCalibrator2 with 512 samples from SQuAD v1.1 as the calibration dataset.

A key finding was that **INT8 ≈ FP16 in throughput at batch ≤ 32 on T4**. Nsight Systems profiling confirmed the reason: both FP16 and INT8 use the same `trt_turing_h1688gemm` Tensor Core kernel, and at these batch sizes the workload is memory-bandwidth-bound rather than compute-bound. The bottleneck is HBM bandwidth, not arithmetic throughput.

| Precision | Kernels % | Memory % | Primary Kernel |
|-----------|-----------|----------|----------------|
| FP32 | 99.4% | 0.6% | trt_turing_sgemm (FP32 GEMM) |
| FP16 | 99.2% | 0.8% | trt_turing_h1688gemm (Tensor Core) |
| INT8 | 99.1% | 0.9% | trt_turing_h1688gemm (Tensor Core) |

### Custom IPluginV3 GELU Plugins

We implemented two custom plugins using the TensorRT 10.x IPluginV3 interface:

**GeluPluginV3** — single-input GELU activation kernel:
```cuda
__device__ inline float gelu_f32(float x) {
    const float k = 0.7978845608f;
    float tanh_val = tanhf(k * (x + 0.044715f * x * x * x));
    tanh_val = fmaxf(-1.0f, fminf(1.0f, tanh_val));
    return x * 0.5f * (1.0f + tanh_val);
}
```

**GeluBiasPluginV3** — fused bias + GELU kernel that computes `output = GELU(x + bias)` in a single pass, eliminating one global memory read/write cycle per BERT encoder layer.

Both plugins were integrated into BERT's ONNX graph via a custom export pipeline using `torch.autograd.Function` with a `symbolic()` method, embedding the custom ops directly across all 12 encoder layers before TensorRT engine build. Numerical accuracy was validated to **< 1e-3 vs PyTorch GELU reference** (standalone plugin test) and **< 0.03 vs FP16 baseline engine** (end-to-end).

Per NVIDIA's official TensorRT Best Practices Guide, GELU should run at FP32 precision due to numerical stability concerns with FP16 [3]. TensorRT automatically inserts FP16→FP32→FP16 cast nodes around the plugin, allowing the rest of the engine to remain in FP16.

### Multiple Optimization Profiles

TensorRT's optimization profile mechanism allows the engine to select different kernels for different input shape ranges. The default single-profile engine (`opt=(8,128)`) selects kernels optimized for batch=8, which are suboptimal at batch=1.

We built a three-profile engine with dedicated profiles for batch=1, 8, and 32:

```python
for opt_batch in [1, 8, 32]:
    profile = builder.create_optimization_profile()
    profile.set_shape("input_ids",
        min=(1,128), opt=(opt_batch,128), max=(opt_batch,128))
    config.add_optimization_profile(profile)
```

This achieved **27% latency reduction at batch=1** (2.71ms → 1.98ms) by allowing TensorRT to select smaller, lower-latency kernels appropriate for single-sample inference.
