This project implements a single-precision General Matrix Multiplication (GEMM) kernel in CUDA C++ from scratch, progressively optimizing from a naive baseline through shared memory tiling, register blocking, float4 vectorization, and systematic parameter tuning. All kernels were profiled on an NVIDIA A100-SXM4-40GB GPU using NVIDIA Nsight Compute. The final tuned kernel achieved **13.51 TFLOPS — 71% of cuBLAS throughput** for a 4096×4096×4096 matrix multiply.

---

## Background: GEMM, the Roofline Model, and GPU Memory Hierarchy

GEMM ($C = A \times B$) is the foundational operation of deep learning — every linear layer, every attention head, every convolution expressed as im2col reduces to matrix multiplication. cuBLAS achieves near-peak throughput through assembly-level optimizations and hardware-specific tuning, but building a high-performance kernel from first principles is the most direct path to understanding GPU architecture [1].

The **roofline model** [2] provides the theoretical framework for GPU performance analysis. Every kernel operates somewhere in the space defined by two limits: the peak memory bandwidth (bytes/second) and the peak arithmetic throughput (FLOPS/second). A kernel's position is determined by its **arithmetic intensity** — the ratio of floating-point operations to bytes loaded from memory (FLOP/byte). Kernels below the ridge point are memory-bound; those above are compute-bound. The A100 has peak fp32 throughput of 19.5 TFLOPS and HBM2e bandwidth of ~2 TB/s.

The GPU memory hierarchy creates a large performance gap between on-chip and off-chip memory:

| Level | Bandwidth | Latency | Scope |
|---|---|---|---|
| HBM2e (global) | ~2 TB/s | ~500 cycles | All SMs |
| L2 Cache | ~12 TB/s | ~200 cycles | All SMs |
| Shared Memory (SRAM) | ~20 TB/s | ~20 cycles | Per SM |
| Registers | ~80 TB/s | ~1 cycle | Per thread |

Warp-level execution is the key to understanding GPU throughput. A warp is a group of 32 threads that execute in lockstep. When a warp stalls waiting for memory, the scheduler switches to another eligible warp to hide the latency. **Occupancy** — the ratio of active warps to the hardware maximum — determines how much latency hiding is available. High register usage per thread limits occupancy, creating a fundamental trade-off between per-thread arithmetic intensity and warp-level latency hiding [3].

## Optimization Journey

### Stage 1 — Naive Baseline

The naive kernel assigns one output element per thread. Each thread independently iterates over the K dimension, reading directly from global memory (HBM) on every step with no data reuse whatsoever.

```cuda
// One thread → one output element
int row = blockIdx.y * blockDim.y + threadIdx.y;
int col = blockIdx.x * blockDim.x + threadIdx.x;
float sum = 0.0f;
for (int k = 0; k < K; k++)
    sum += A[row * K + k] * B[k * N + col];
C[row * N + col] = sum;
```

Nsight Compute immediately confirms the diagnosis: memory throughput near 70%, compute throughput near 14%, and SM utilization almost zero. The kernel is **deeply memory-bound** — it does almost no arithmetic per byte loaded.

**Result: 2.43 TFLOPS (4096×4096) — 12.8% of cuBLAS**

---

### Stage 2 — Shared Memory Tiling

The standard fix for memory-bound GEMM is shared memory tiling [1, 4]. A thread block cooperatively loads a BK-wide tile of A and B from HBM into shared memory (`sA`, `sB`), then all threads compute their partial dot products locally before loading the next tile. Each float loaded from HBM now contributes to `BM × BN` multiply-adds rather than one, multiplying arithmetic intensity by the tile size.

```
Arithmetic intensity per tile:
  Compute:    BM × BN × BK × 2  FLOP
  Load:      (BM × BK + BK × BN) × 4  bytes
  Intensity = (BM × BN × BK × 2) / ((BM + BN) × BK × 4)
            ≈ BM/8  FLOP/byte  (for BM = BN)
```

For BM = BN = 64, intensity ≈ 8 FLOP/byte — already 50× better than naive.

**Result: 5.33 TFLOPS — 2.2× over naive**

---

### Stage 3 — Register Blocking

Shared memory tiling alone still has each thread computing one output element. Register blocking has each thread compute a `TM × TN` submatrix (we use TM = TN = 8), accumulating into register arrays `regC[TM][TN]`. The outer product loop loads one column of A and one row of B from shared memory into registers (`regA[TM]`, `regB[TN]`), then computes 64 multiply-adds without touching shared memory again.

```cuda
// Each thread owns 8×8 = 64 output elements
float regC[TM][TN] = {0.0f};

for (int kk = 0; kk < BK; kk++) {
    float regA[TM], regB[TN];
    for (int m = 0; m < TM; m++) regA[m] = sA[threadRow*TM + m][kk];
    for (int n = 0; n < TN; n++) regB[n] = sB[kk][threadCol*TN + n];
    for (int m = 0; m < TM; m++)
        for (int n = 0; n < TN; n++)
            regC[m][n] += regA[m] * regB[n];
}
```

The key insight: `regA` and `regB` are each loaded once from shared memory per `kk` step, but participate in `TM × TN = 64` multiply-adds. Arithmetic intensity at the shared memory level doubles again.

The cost is register pressure: `regC[8][8]` consumes 64 registers per thread, limiting theoretical occupancy to 25% on A100 (4 active warps per scheduler vs. the hardware maximum of 16).

**Result: 10.62 TFLOPS — 4.4× over naive**

---

### Stage 4 — Float4 Vectorized Loads

Replacing scalar `float` global memory accesses with `float4` (128-bit) loads and stores maximizes the HBM2e bus utilization. Each load instruction now moves 4 floats instead of 1, reducing instruction count by 4× for the same data transfer.

```cuda
// Load 4 consecutive floats in one 128-bit transaction
float4 tmp = reinterpret_cast<const float4*>(&A[globalRow * K + globalCol])[0];
sA[row][col]     = tmp.x;
sA[row][col + 1] = tmp.y;
sA[row][col + 2] = tmp.z;
sA[row][col + 3] = tmp.w;
```

The stride loop iterates in units of `float4` groups (`i < (BM * BK) / 4`), with each thread responsible for loading non-overlapping groups across the tile. Boundary conditions fall back to scalar loads.

**Result: 12.40 TFLOPS — 5.1× over naive**

### Stage 5 — Systematic Parameter Tuning

With the core optimization structure in place, the final gains come from tuning two parameters that directly control the trade-offs identified in the profile: **BK size** (which governs how much K-dimension work is done per tile load) and **shared memory padding** (which eliminates bank conflicts).

#### 5a — Tuning BK: Reducing Synchronization Overhead

The outer loop over the K dimension iterates `K / BK` times, and each iteration ends with `__syncthreads()` — a barrier that serializes all threads in the block. Every synchronization is dead time.

Doubling BK from 8 to 16 halves the number of `__syncthreads()` calls from `K/8 = 512` to `K/16 = 256` for K = 4096. It also means each float4 load stride loop runs more iterations per thread (`(BM × BK) / 4 / totalThreads` increases), which amortizes the loop overhead over more useful work.

However, BK cannot grow unboundedly. Shared memory per block scales as:

```
sA = BM × (BK + padding) × 4 bytes
sB = BK × (BN + padding) × 4 bytes

BK=8:  sA + sB ≈ 5 KB   → 32 blocks/SM resident (theoretical)
BK=16: sA + sB ≈ 9 KB   → 18 blocks/SM resident
BK=32: sA + sB ≈ 17 KB  → 9  blocks/SM resident
```

At BK=32, the increased shared memory footprint reduces the number of resident blocks per SM, hurting occupancy and erasing the synchronization gains.

| BK | Sync calls (K=4096) | Shared Mem | 4096 TFLOPS |
|---|---|---|---|
| 8  | 512 | ~5 KB  | 10.62 |
| 16 | 256 | ~9 KB  | 13.51 |
| 32 | 128 | ~17 KB | 10.99 |

**BK = 16 is the sweet spot** — synchronization overhead is cut in half without significantly degrading occupancy.

#### 5b — Tuning Padding: Eliminating Bank Conflicts

Shared memory on NVIDIA GPUs is organized into 32 banks, each 4 bytes wide. Address `a` maps to bank `a % 32`. When multiple threads in a warp access the same bank simultaneously, their requests serialize — a **bank conflict**.

In a tiled GEMM, the outer product loop reads `sA[threadRow * TM + m][kk]` for `m = 0..TM-1`. All threads in a column read the same column `kk` of `sA`, stepping down rows. With BK=16 and no padding, row `r` of `sA` starts at address `r × BK = r × 16`, mapping to bank `(r × 16) % 32`. Rows 0 and 2 both start at bank 0, rows 1 and 3 at bank 16 — only two distinct starting banks for all 64 rows, causing heavy conflicts.

Adding `padding` extra floats per row shifts each row's starting address:

```
Row r starts at bank: (r × (BK + padding)) % 32

BK=16, padding=0: row 0→bank 0, row 1→bank 16, row 2→bank 0 ...  (2-way conflict)
BK=16, padding=1: row 0→bank 0, row 1→bank 17, row 2→bank 2 ...  (distributed, better)
BK=16, padding=2: row 0→bank 0, row 1→bank 18, row 2→bank 4 ...  (evenly distributed)
BK=16, padding=4: row 0→bank 0, row 1→bank 20, row 2→bank 8 ...  (partially distributed)
```

Padding=2 makes consecutive rows land on distinct banks modulo 32, nearly eliminating conflicts for this tile geometry.

## Full Performance Summary

All results are 10-run averages after a warmup pass. Benchmarks run on NVIDIA A100-SXM4-40GB, sm_80.

| Implementation | 1024 (TFLOPS) | 2048 (TFLOPS) | 4096 (TFLOPS) | vs cuBLAS |
|---|---|---|---|---|
| 01_gemm_naive         | 2.86  | 3.16  | 2.43  | 12.8% |
| 02_gemm_tiled         | 5.13  | 5.41  | 5.33  | 28.0% |
| 03_gemm_regblock      | 5.62  | 7.97  | 10.62 | 55.8% |
| 04_gemm_float4        | 5.96  | 9.24  | 12.40 | 65.2% |
| **tuning/BK16_pad2**  | **6.88** | **10.62** | **13.51** | **71.0%** |
| 05_gemm_cublas        | 16.38 | 17.60 | 19.02 | 100%  |

Notable: `05_gemm_float4_ultimate` (11.37 TFLOPS) underperforms `04_gemm_float4` (12.40 TFLOPS) at 4096 despite adding `__launch_bounds__` and padding. Nsight confirmed the cause: `__launch_bounds__(256, 2)` forced the compiler to spill registers into local memory, turning a compute-bound kernel into a memory-bound one and raising bank conflict rates from 20% back to 41%.

## What's Left: The Gap to cuBLAS

The remaining 29% gap between the best custom kernel (13.51 TFLOPS) and cuBLAS (19.02 TFLOPS) is attributable to three factors, in approximate order of impact:

**1. No latency hiding from low occupancy.** With only ~26% occupancy (~8 active warps per SM), there are insufficient warps to cover the ~500-cycle global memory latency. The long-scoreboard stall rate of 37.0% in the profile directly reflects this. *Fix: double buffering — overlap the next tile's global load with the current tile's computation using two alternating shared memory buffers.*

**2. Residual bank conflicts.** Even with padding=2, ~20% of shared memory transactions are still serialized. The root cause is that the outer product loop reads `sA` and `sB` in patterns that are difficult to fully de-conflict without restructuring the thread layout. *Fix: warp swizzling or transposing one of the shared memory tiles.*

**3. No Tensor Core utilization.** cuBLAS uses WMMA (Warp Matrix Multiply-Accumulate) instructions that execute 16×16×16 matrix multiply in a single instruction at dramatically higher throughput. All custom kernels here use scalar FP32 FMAs. *Fix: restructure to `wmma::mma_sync` — a fundamentally different kernel design.*
