This project implemented a General Matrix Multiplication (GEMM) kernel in CUDA C++ from scratch, progressively optimizing from a naive baseline through shared memory tiling, register blocking, and float4 vectorization. All kernels were profiled on an NVIDIA A100-SXM4-40GB GPU using NVIDIA Nsight Compute. The final kernel achieved over 80% of cuBLAS throughput at 7.60 TFLOPS.

## Background: GEMM, the Roofline Model, and GPU Memory Hierarchy

GEMM ($C = A \times B$) is the foundational operation of deep learning — every linear layer, every attention head, every convolution expressed as im2col reduces to matrix multiplication. cuBLAS achieves near-peak throughput through assembly-level optimizations and hardware-specific tuning, but building a high-performance kernel from first principles is the most direct path to understanding GPU architecture [1].

The **roofline model** [2] provides the theoretical framework for GPU performance analysis. Every kernel operates somewhere in the space defined by two limits: the peak memory bandwidth (in bytes/second) and the peak arithmetic throughput (in FLOPS/second). A kernel's position is determined by its **arithmetic intensity** — the ratio of floating-point operations to bytes loaded from memory (FLOP/byte). Kernels below the "ridge point" (the intensity at which compute throughput equals memory bandwidth) are memory-bound; those above are compute-bound. The A100 has peak fp32 throughput of 312 TFLOPS and HBM bandwidth of ~2 TB/s, giving a ridge point of approximately 156 FLOP/byte.

The GPU memory hierarchy creates a large performance gap between on-chip and off-chip memory. HBM (High Bandwidth Memory) offers ~2 TB/s bandwidth but has high latency and is shared across all SMs. L1/shared memory per SM is ~192 KB at full throughput and much faster. Registers are fastest but strictly limited — the A100 has 65,536 32-bit registers per SM, shared among all resident warps [3].

Warp-level execution is the key to understanding GPU throughput. A warp is a group of 32 threads that execute in lockstep. Warps stall when waiting for memory or ALU results; the scheduler hides this latency by switching to another eligible warp. **Occupancy** — the ratio of active warps to the hardware maximum — determines how much latency hiding is available. High register usage per thread limits occupancy, creating a fundamental trade-off between per-thread arithmetic intensity and warp-level latency hiding [3].

## What We Did

**Naive baseline.** The naive kernel assigns one thread per output element; each thread reads an entire row of A and column of B directly from HBM, with no reuse. Nsight Compute immediately confirmed the diagnosis: low L2 cache hit rate, HBM throughput near saturation, and SM utilization near zero. The kernel delivered **4.08 TFLOPS** — about 1.3% of A100 peak.

**Shared memory tiling.** The standard fix for memory-bound GEMM is shared memory tiling [1, 4]. A thread block cooperatively loads a tile of A and a tile of B from HBM into shared memory, then all threads compute their partial dot products from shared memory before loading the next tile. Each element loaded from HBM now contributes to the full tile's worth of computation rather than a single multiply-add, increasing arithmetic intensity. Result: **5.08 TFLOPS**.

Nsight flagged a 50% shared memory bank conflict rate across 53.6 million load requests. Shared memory on NVIDIA GPUs is organized into 32 banks; simultaneous accesses to the same bank by threads in a warp serialize [3]. Memory padding — adding one extra column to the shared memory tile layout — shifted the access stride so that consecutive thread accesses landed on consecutive banks, eliminating conflicts.

**Register blocking.** Register blocking has each thread compute a small $r \times r$ submatrix of the output (we used $r = 4$) rather than a single element [4]. This increases the amount of arithmetic done per memory load by a factor of $r$, pushing the kernel further toward compute-bound on the roofline. Result: **7.35 TFLOPS**.

The cost was significant: 95 registers per thread, capping theoretical occupancy at 25% (4 active warps per scheduler versus the hardware maximum of 16). This is the occupancy/intensity trade-off in practice.

**Float4 vectorization.** Replacing scalar `float` memory accesses with `float4` loads and stores coalesces four consecutive floats into a single 128-bit transaction, maximizing memory bus utilization on A100's HBM2e. Result: **7.60 TFLOPS** — over 80% of cuBLAS throughput.

The remaining gap to 100% cuBLAS throughput comes from architectural details not accessible in vanilla CUDA C++: Tensor Core WMMA instructions, assembly-level pipeline scheduling, and hardware-specific memory access patterns that require intimate microarchitectural knowledge.
